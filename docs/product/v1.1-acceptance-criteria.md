# LeanMate V1.1 验收标准与业务边界

Last updated: 2026-06-07

## 文档定位

本文用于把 V1.1 PRD、OpenAPI 和 iOS 设计稿之间的业务边界落成可验收规则。

本文不新增 V1.1 功能范围。业务开发仍以以下文档为准：

- `docs/product/prd-v1.1.md`
- `docs/api/openapi.yaml`
- `design/app/LeanMateV1.0-shikaka.pen`
- `ios/docs/design-screen-map.md`

## 验收原则

- 每个页面状态都必须有明确的下一步行动。
- 能用页面内校验解决的问题，不等接口失败后再提示。
- 用户输入、AI 识别、保存记录都要允许失败后继续完成核心任务。
- AI 输出只能作为减脂建议和记录辅助，不做健康诊断、疾病建议或医疗结论。
- OpenAPI 是客户端和服务端的数据契约；客户端不得实现 OpenAPI 之外的接口和业务能力。

## 全局验收标准

### 登录与游客态

V1.1 首发真实登录只支持 Apple 登录。local/test 环境可以使用受控 mock verifier。

游客态验收：

- 用户可以进入欢迎页和有限的游客态首页提示。
- 游客不能保存长期数据，不能保存用户档案、饮食记录、体重记录或日报查看状态。
- 游客点击需要持久化的动作时，引导登录，不直接丢失用户已输入内容。
- 登录成功后，如果 `profileCompleted = false`，进入档案填写；如果 `profileCompleted = true`，进入首页。

开发处理：

- iOS 路由以 Token 状态和 `profileCompleted` 为主，不依赖页面自己猜测。
- `MockAPIClient` 必须覆盖游客、档案未完成和已完成档案三类入口状态。

### 用户档案填写规则

档案字段必须符合 `SaveUserProfileRequest`：

- `gender`: `male` / `female` / `unknown`
- `age`: 1-120
- `heightCm`: 50-250
- `currentWeightKg`: 20-300
- `targetWeightKg`: 20-300
- `activityLevel`: `sedentary` / `light` / `moderate` / `active` / `very_active`
- `timezone`: 必填
- `targetDate`: 可选

体验验收：

- 表单在本地即时校验，错误提示必须具体。
- 字段合法但明显不符合核心用户画像时，可以用提示文案引导，但不额外扩大硬性限制。
- 保存成功后展示后端返回的 `bmi`、`bmrKcal` 和 `dailyCalorieTargetKcal`。
- V1.1 不提供手动修改每日热量目标能力，目标以后端计算结果为准。

开发处理：

- iOS ViewModel 负责本地校验和保存状态。
- 后端仍保留最终校验，客户端不能只依赖本地校验。

### 目标生成

目标生成验收：

- 保存档案成功后必须生成并返回 BMI、BMR 和每日推荐热量目标。
- 首页和我的页展示的目标值以后端返回为准。
- 用户重新保存档案后，目标值允许随档案变化重新计算。

开发处理：

- 客户端可以展示后端返回结果，但不独立计算并覆盖服务端结果。
- 目标解释保持简洁，不做复杂计算过程页面。

### 日期与时区

业务日期验收：

- 业务日期按用户 `timezone` 计算。
- 服务端存储时间使用 UTC。
- 首页、饮食、体重、日报和连续打卡按业务日期聚合。
- 接口日期参数统一使用 `yyyy-MM-dd`。

补录验收：

- `mealDate` 和 `recordDate` 支持传入非今天日期。
- V1.1 不扩展复杂日历能力；设计稿没有的日历筛选不在首批实现范围。
- 补录记录应按传入业务日期影响首页统计和连续打卡。

开发处理：

- iOS 不直接用设备日期推断后端业务聚合结果。
- 进入页面时使用后端返回的 `date` 和用户 timezone 展示“今天”等文案。

### 首页状态

首页必须覆盖以下状态：

| 状态 | 验收结果 | 主行动 |
| --- | --- | --- |
| loading | 展示加载状态，不阻塞导航 | 等待或返回 |
| visitor | 展示游客提示 | 登录 |
| profileIncomplete | 提示完成档案 | 去填写档案 |
| empty | 没有饮食和体重记录时展示空态 | 去记录 |
| loaded | 展示目标热量、摄入热量、剩余热量、体重、连续打卡、饮食记录、日报摘要 | 继续记录或查看日报 |
| error | 展示可理解错误 | 重试 |

开发处理：

- 每个状态只保留一个最强主行动，避免空态里堆多个同级入口。
- 空态和错误态优先复用 `LMStateView`。

### 饮食记录与 AI 识别

记录方式验收：

- 拍照、文本和手动录入都必须进入确认流程。
- 用户确认前不计入正式饮食记录。
- 用户可以编辑食物名称、数量、重量、热量和营养信息。
- 保存成功后更新今日热量统计和营养统计。

AI 识别状态验收：

- `pending` / `running`：展示识别中状态，允许用户返回或取消等待。
- `succeeded`：展示可编辑确认页。
- `failed`：展示失败原因，并允许用户改为手动编辑继续保存。

开发处理：

- 客户端轮询识别任务时必须可取消，不能造成重复保存。
- 确认页保存按钮需要防重复点击。
- 草稿内容优先保留到 `LocalStore`，避免识别失败或网络失败导致用户重新输入。

### AI 日报

日报验收：

- 日报状态覆盖 `pending` / `generated` / `viewed` / `failed`。
- 日报内容为 3-5 句简洁、可执行建议。
- 没有足够记录时，不生成空洞日报，提示用户先完成饮食或体重记录。
- 生成失败时允许重试。
- 查看日报后调用 `POST /v1/reports/daily/{reportId}/view` 标记已读。

开发处理：

- 日报页面不要直接展示 AI 原始错误。
- 日报失败不影响用户继续记录饮食和体重。

### 连续打卡

V1.1 连续打卡口径：

- 用户在某个业务日期有 confirmed 饮食记录，或有体重记录，即视为该业务日期已打卡。
- 补录饮食或体重记录时，应按记录所属业务日期重新计算连续打卡。
- 里程碑只展示 OpenAPI 支持的 3、7、30、100 天。

开发处理：

- 首页展示 `currentDays`，我的页或里程碑弹窗展示 `longestDays` 和 milestone。
- 客户端不自行计算 streak，只展示后端返回结果。

### 错误、离线与重试

通用错误验收：

- 401：清理失效访问态并引导重新登录。
- 400：展示具体字段或请求错误。
- 403/404：展示资源不可访问或不存在。
- 500 / AI 服务错误：展示可重试提示，不阻断用户继续记录。

离线验收：

- V1.1 不做完整离线同步功能。
- 正在编辑的饮食草稿和待保存体重可以保存在本地，恢复网络后由用户主动重试。
- 不在用户不知情的情况下自动创建重复记录。

开发处理：

- ViewModel 状态必须区分 idle、loading、success、empty、error。
- 保存类动作必须处理重复点击和接口超时。

## 模块验收标准

### Profile

- 未完成档案的登录用户必须先进入档案填写流程。
- 保存档案成功后进入首页。
- 保存失败时保留已输入内容。
- 字段校验错误在表单内展示。

### Home

- 展示 `GET /v1/home/today` 返回的今日状态。
- 档案未完成时，不展示假的热量目标。
- 空态必须引导用户完成第一条记录。
- 首页不直接生成 OpenAPI 之外的统计。

### Diet

- 三种记录方式都必须能到达确认页。
- AI 识别失败后仍能转为手动记录。
- 删除饮食记录前必须二次确认。
- 保存、编辑、删除成功后刷新首页或本页统计。

### Weight

- 支持记录业务日期和体重。
- 体重输入范围按 OpenAPI 的 20-300 kg 校验。
- 保存失败时保留输入内容。
- 7 天和 30 天趋势只在数据可用时展示；数据不足时展示空态。

### Report

- 可获取、生成、查看日报。
- 生成中、失败、已生成、已查看状态都要覆盖。
- 日报建议必须短、具体、可执行。

### Retention

- 首页展示当前连续天数。
- 达成里程碑时可展示弹窗。
- 里程碑弹窗不阻断记录流程。

## 额外需要考虑的 V1.1 边界

以下不是新增主功能，但会影响体验和验收。

### 隐私与权限

- 拍照或相册权限只在用户选择拍照记录时请求。
- 图片上传、保存期限和删除策略需要在后端/隐私文档中继续确认。
- AI 识别结果需要提示为估算值，避免用户把它当作绝对精确数据。

### 埋点与指标

- 核心成功指标依赖 `docs/data/events.md` 和 `docs/data/metrics.md`。
- 首次饮食、首次体重、日报查看、连续 3/7 天必须能被事件验证。
- 服务端确认类事件优先作为核心指标口径。

### 可访问性与文案

- 主要按钮、输入框和状态提示要支持较长中文文案，不得截断关键动作。
- 错误文案用用户能理解的语言，不暴露内部错误码。
- 颜色不能作为唯一状态表达，关键状态需有文字或图标辅助。

### 数据一致性

- 保存类接口返回后，客户端以服务端返回结果刷新本地状态。
- 重复点击、网络重试和页面返回不能创建重复记录。
- 本地草稿只作为恢复输入使用，不作为服务端事实来源。

### 发布配置

- Bundle ID、Apple Developer Team、Sign in with Apple 配置需要在接真实登录前确认。
- dev/local/prod API Base URL 需要通过环境配置切换，不写死在业务页面。

## 开发流程要求

每个业务模块开工前先补一张状态矩阵，至少包含：

| 状态 | 用户看到什么 | 用户能做什么 | 接口 | Mock 场景 | 验收点 |
| --- | --- | --- | --- | --- | --- |

推荐开发顺序：

1. 写清模块状态矩阵。
2. 用 `MockAPIClient` 覆盖 success、loading、empty、error、visitor、profileIncomplete。
3. 实现 ViewModel 状态流转。
4. 实现 SwiftUI 页面。
5. 补 ViewModel 或基础交互测试。
6. 再接真实后端。

这样可以在页面早期解决游客态、空态、失败态和补录边界，减少后期返工。
