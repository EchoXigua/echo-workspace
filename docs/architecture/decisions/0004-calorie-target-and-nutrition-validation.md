# ADR 0004：热量目标校准和饮食营养数据校验策略

## 状态

已接受

## 背景

LeanMate V1.1 需要基于用户档案给出每日推荐热量目标，并根据饮食记录计算当日摄入和剩余热量。

初始每日目标热量可以通过 BMI、BMR、TDEE 和减脂缺口公式估算，但该结果只是一种起点，会受到活动水平、真实代谢、肌肉量、饮食记录准确度和体重波动影响。饮食记录侧，AI 识别和用户手动录入都可能产生不准确的食物重量、热量或三大营养素数据。

因此需要明确两件事：

- 热量目标后续如何根据真实体重趋势调整。
- 客户端传入的食物营养数据如何进入统计。

## 决策

### 热量目标校准

V1.1 当前接口仍按用户档案公式计算并保存 `dailyCalorieTargetKcal`。后续热量目标动态调整作为建议能力，不在后台自动修改用户目标。

建议能力的产品口径：

- 使用 7-14 天体重趋势判断当前目标是否偏高或偏低。
- 使用 7 日移动平均体重或 14 天趋势，避免单日水分、盐分、排便等波动误导判断。
- 若体重下降慢于预期，建议每日目标小幅下调，例如 `-100` 到 `-150` kcal。
- 若体重下降过快，建议每日目标小幅上调，例如 `+100` 到 `+150` kcal。
- 建议必须展示原因，并要求用户确认后才更新目标。
- 不因为趋势判断自动覆盖用户档案或历史目标快照。

建议响应未来可以表达为：

```json
{
  "currentTargetKcal": 1800,
  "suggestedTargetKcal": 1700,
  "changeKcal": -100,
  "reason": "过去 14 天体重下降慢于预期",
  "requiresUserConfirmation": true
}
```

### 饮食营养数据

客户端可以传入用户确认后的食物项、重量、热量和营养素。后端不能完全信任这些数据，必须做合理性校验后再进入统计。

后端职责：

- 只接受食物项明细，不接受客户端传入整餐总热量。
- 整餐总热量、蛋白质、脂肪、碳水由后端根据食物项汇总。
- 对食物项的热量、重量、蛋白质、脂肪、碳水做硬性范围校验。
- 对宏量营养素和热量做一致性校验，例如 `proteinG * 4 + carbsG * 4 + fatG * 9` 与 `caloriesKcal` 偏差不能过大。
- 当 `weightG` 存在时，对每 100g 热量做合理范围校验；极端值应拒绝或要求用户确认。
- 保存或推导营养数据来源和可信度，便于后续质量分析。

营养数据来源目标枚举：

```text
ai_estimated      AI 估算
food_db           食物库匹配
user_confirmed    用户确认
user_override     用户手动改过
```

食物热量查询和食物库不在本决策中展开，后续单独设计。

## 后果

优点：

- 初始热量目标可立即使用，后续还能根据真实数据逐步个性化。
- 不自动修改目标，避免用户困惑和错误调整。
- 允许 AI 和用户修正参与记录流程，同时通过后端校验避免离谱数据污染统计。
- 为后续食物库、AI 估算质量评估和个性化推荐预留来源口径。

代价：

- 后端需要补充营养合理性校验规则和测试。
- 如果未来增加 `nutritionSource` 等字段，需要同步 OpenAPI、数据库和客户端模型。
- 食物库和外部热量查询仍需单独设计，不应混在饮食保存接口里临时实现。

## 已落地能力

- `GET /v1/profile/calorie-target-suggestion` 提供热量目标调整建议，不自动修改目标。
- 饮食记录保存、编辑和游客本地记录同步路径已接入营养数据合理性校验。

## 后续动作

- 评估是否在 `food_items` 增加营养来源字段。
- 单独设计食物热量查询、食物库、AI 估算和用户修正之间的数据流。
