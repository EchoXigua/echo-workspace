# ADR 0003：V1.1 暂不引入 Redis 和 MQ

## 状态

已接受

## 背景

LeanMate V1.1 需要后端支持用户档案、饮食记录、体重记录、每日统计、AI 日报和连续打卡。AI 图片识别、文本解析、日报生成存在异步处理需求，因此需要评估是否一开始引入 Redis 和消息队列。

## 决策

V1.1 暂不引入 Redis 和 MQ。

使用 PostgreSQL 作为核心数据源，使用数据库任务表 + Spring Scheduler 或后台 worker 处理早期异步任务。

## 原因

- 当前阶段核心目标是验证产品价值，不是优化高并发基础设施。
- PostgreSQL 足以承载 V1.1 的核心业务读写和任务状态管理。
- Redis 和 MQ 会增加部署、监控、排障和一致性成本。
- AI 任务状态保存在数据库中，便于查询、重试和问题追踪。

## 后果

优点：

- 本地开发和部署简单。
- 基础设施少，维护成本低。
- 任务状态可直接在数据库中排查。

代价：

- 后台任务吞吐能力有限。
- 数据库需要承担任务扫描和状态更新。
- 当任务量增长后，需要再拆 worker 或引入 MQ。

## 后续触发条件

满足以下条件时重新评估 Redis：

- 需要验证码、限流、短期 token 黑名单。
- 首页或统计接口读压力明显增加。
- 需要分布式锁或热点缓存。

满足以下条件时重新评估 MQ：

- AI 识别、日报、周报、月报任务量明显增加。
- 后台任务影响主 API 响应。
- 需要多个 worker 横向扩容。
- 行为事件采集量大，需要异步分析链路。
