# LeanMate 版本处理流程

处理新的 LeanMate PRD 或版本规划时使用本流程。

## 1. 接收输入

读取用户提供的文档，识别：

- 版本号；
- 产品目标；
- 新增功能；
- 行为变化；
- 移除或延期范围；
- 成功指标；
- 影响哪些端；
- 对后端、API、数据、AI 的影响。

如果源文件在仓库外，先复制到 `docs/product/`，并使用稳定文件名。

## 2. 梳理产品变化

和当前 PRD、roadmap 对比，整理：

- 新增范围；
- 保持不变的范围；
- 明确不做的范围；
- 依赖项；
- 待确认问题。

产品层面的结论写进产品文档，不要藏在后端文档里。

## 3. 判断架构影响

只有当 PRD 改变以下内容时，才更新架构文档：

- 领域模型；
- 后端数据归属；
- 客户端和后端职责；
- AI 流程；
- 存储或数据保留策略；
- 同步或离线策略；
- 埋点、指标或分析链路。

对以后可能反复讨论的重要技术取舍，写 ADR。

## 4. 编写后端技术方案

创建版本目录：

```text
server/docs/technical-design/vX.Y/
├── README.md
└── NN-feature-name.md
```

每个需求方案至少包含：

- PRD 对应章节；
- 后端职责；
- 不做范围；
- 核心流程；
- 数据模型影响；
- API 影响；
- 状态和业务规则；
- 异常和降级；
- 埋点和指标；
- 测试要点；
- 待确认问题。

使用 `server/docs/technical-design/template.md`。

## 5. 更新 OpenAPI

后端技术方案明确后，更新 `docs/api/openapi.yaml`：

- 新增或修改 paths；
- 新增或修改 schemas；
- 标注必填字段；
- 明确状态枚举；
- 必要时补充错误响应；
- 除非版本明确破坏兼容，否则保持向后兼容。

编辑后校验 YAML 语法。

## 6. 判断是否可以并行开发

满足以下条件时，后端和 iOS/Android/Web 可以并行：

- 功能范围明确；
- 后端技术方案已存在；
- OpenAPI 已包含所需接口和 schema；
- 待确认问题已列出；
- `docs/project-state.md` 写清下一步。

## 7. 会话交接

每次重要工作结束都更新 `docs/project-state.md`：

- 最后更新时间；
- 当前版本焦点；
- 已完成产物；
- 已接受决策；
- 已知缺口；
- 下一步；
- 新对话恢复提示。
