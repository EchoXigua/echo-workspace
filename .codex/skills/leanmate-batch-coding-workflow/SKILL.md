# LeanMate 批次编码工作流

## 核心目标

用一个轻量主线程做长期调度，把具体编码、调试、构建和提交放到独立子线程里完成。每个批次结束时把关键状态写入仓库文档和 Git commit，避免依赖聊天历史恢复项目状态。

## 角色分工

### 主线程

只做项目级调度和收口：

- 确认需求是否进入编码环节。
- 拆分批次和明确每批边界。
- 创建或派生子线程，并发送启动提示。
- 记录每批结果：commit hash、验证结果、遗留问题、下一批建议。
- 避免承载大量代码 diff、构建日志和调试过程。

### 子线程

负责单个批次的完整工程执行：

- 读取项目规则、项目状态、PRD、OpenAPI、验收标准和当前端/模块文档。
- 检查 `git status --short`，保护已有未提交改动。
- 实现本批功能，不扩展到下一批。
- 补必要测试。
- 运行本批要求的构建和测试。
- 更新必要状态文档。
- 在用户要求时按 `.codex/skills/leanmate-git-commit/SKILL.md` 提交。

## 线程选择

优先按目标选择：

- 需要继承当前主线程的项目级背景，但不想污染主线程：使用派生线程。
- 需要尽量干净的上下文，只依赖仓库文档和启动提示：创建新项目线程。
- 需要隔离文件改动或并行开发不同批次：使用 worktree 线程。
- 同目录子线程正在运行时，主线程不要编辑同一批次文件。

如果工作区已有未提交改动，必须在启动提示中说明哪些改动不能碰。

## 批次启动流程

1. 先在主线程定义本批目标、范围和禁止事项。
2. 要求子线程先读：
   - `AGENTS.md`
   - 对应端目录的 `AGENTS.md`
   - `docs/project-state.md`
   - 当前版本 PRD、验收标准和 OpenAPI
   - 对应端或模块的状态、架构、设计映射文档
3. 要求子线程先运行 `git status --short`。
4. 明确只提交或只修改本批相关文件。
5. 明确完成后的验证命令。
6. 明确是否自动提交；默认不要自动提交，除非用户要求。

启动提示模板见 `references/batch-thread-prompts.md`。

## 批次收口流程

每个子线程结束前应输出或落盘：

- 本批完成范围。
- 修改文件或模块摘要。
- 验证命令和结果。
- 未能运行的测试及原因。
- 是否提交，若已提交则记录 commit hash。
- 下一批建议。

重要编码批次应更新 `docs/project-state.md`，至少包含：

- 最新完成批次。
- 最新 commit hash。
- 当前已知缺口。
- 下一批推荐启动提示。

如果该批新开了模块状态矩阵，应同时更新对应状态矩阵文档，例如 `ios/docs/v1.1-state-matrix.md`。

## 提交原则

- 每批通常一个提交，除非批次内部包含明显独立功能。
- 提交前只 stage 本批文件，不带入其他线程或用户的脏改动。
- 使用项目内提交规范 skill：`.codex/skills/leanmate-git-commit/SKILL.md`。
- 提交信息使用中文 Conventional Commits，例如：
  - `feat(ios): 实现首页今日状态`
  - `feat(server): 实现饮食识别任务接口`
  - `docs(docs): 更新项目状态`

## 适用提醒

当用户说“新需求进入编码环节”“后续按批次开发”“新开对话跑下一批”“帮我启动第 N 批”“把上下文和进度落盘”时，使用本 skill。

不要把本 skill 当成具体业务实现规范。具体业务仍以 PRD、OpenAPI、验收标准、端侧文档和设计稿为准。
